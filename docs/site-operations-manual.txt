KK Archive - Website Operations Manual

1. Overview

This project is managed through four main services:

- GitHub: source code and version history
- Vercel: website deployment
- Supabase: production database
- Cloudflare R2: image storage

Daily content management and feature development are separate workflows.


2. Daily Content Workflow

When new content is ready:

1. Put reviewed folders into:
   db image/clean

2. For normal day-to-day updates, run:
   npm run sync:new

This will:

- enrich missing Pixiv metadata
- upload new images to R2
- rebuild post.json only for new folders
- import new content into the database

3. If you need to rebuild all clean content from scratch, run:

   npm run sync:all

This full rebuild runs the same steps in this order:

   npm run clean:manifest
   npm run clean:pixiv
   npm run r2:upload-clean
   npm run clean:post-json
   npm run clean:import-all

Notes:

- clean:manifest does not need to run twice
- clean:post-json only writes post.json files
- clean:post-json:new only writes post.json for folders not yet imported
- sync:new is a short alias for clean:sync-new
- sync:all is a short alias for clean:sync-all
- database import happens in clean:import-new or clean:import-all

Optional cleanup:

If you want to quickly move folders that are still not uploaded/imported out of
`db image/clean` into `db image/later`:

1. Preview candidates:
   npm run clean:move-unuploaded:later

2. Apply move:
   npm run clean:move-unuploaded:later:apply

This avoids manual one-by-one checks.

4. Open the website admin:

- /admin
- /admin/contents

5. Review flow:

- AUDIT role:
  Unverified -> Edited

- ADMIN role:
  Edited -> Passed


3. Content Review Rules

Review states:

- Unverified:
  raw imported content, not yet reviewed

- Edited:
  reviewed and adjusted by audit staff

- Passed:
  approved by admin

Public users should not rely on internal review labels except where the UI explicitly shows Unverified warnings.


4. Account and Risk Management

Use:

- /admin
- /admin/activity

Available controls:

- suspend suspicious accounts
- inspect login activity
- review member activity signals
- manage role changes

Current role structure:

- MEMBER
- AUDIT
- ADMIN


5. Local Development Workflow

If you want to change the website UI, features, or logic:

1. Work locally on your own machine
2. Test locally
3. Commit changes to git
4. Push to GitHub
5. Let Vercel deploy the new version

Recommended local process:

1. Start local development:
   npm run dev

2. If you changed production-related database behavior, verify build:
   npm run build:postgres

3. Commit:
   git add .
   git commit -m "Describe the change"

4. Push:
   git push

Vercel will then deploy from GitHub.


6. Production Deployment Workflow

Main deployment path:

- code -> GitHub
- build -> Vercel
- database -> Supabase
- images -> R2

Important production environment variables:

- POSTGRES_POOLED_URL
- POSTGRES_SESSION_URL
- SESSION_SECRET
- GOOGLE_CLIENT_ID
- GOOGLE_CLIENT_SECRET
- GOOGLE_REDIRECT_URI
- R2_ACCOUNT_ID
- R2_ACCESS_KEY_ID
- R2_SECRET_ACCESS_KEY
- R2_BUCKET_NAME
- R2_PUBLIC_BASE_URL


7. Database Migration Notes

Local development may still use SQLite for convenience.

Production uses Supabase Postgres.

Useful commands:

- npm run db:generate:postgres
- npm run db:push:postgres
- npm run db:migrate:sqlite-to-postgres
- npm run db:use:postgres
- npm run build:postgres


8. Backup Recommendations

At minimum, keep backups of:

- Supabase database exports
- R2 bucket contents or original local content source
- db image/output if it remains your raw source archive
- db image/clean if it remains your reviewed import source

Do not rely only on the deployed website as your backup.


9. Security Checklist

Before or after production updates, rotate sensitive secrets when needed:

- Supabase database password
- Google OAuth client secret
- Cloudflare R2 keys

Never commit these files:

- .env
- local session files
- local database files
- large raw image archives unless intentionally needed
